01 · Introduction

Background

What is a NIDS?

A Network Intrusion Detection System (NIDS) monitors incoming network traffic and classifies it as benign or malicious. When malicious traffic is detected, the system takes action; otherwise, it allows access.

Incoming Traffic
→
IDS Model
→
Prediction
→
Take Action
Benign → Allow Access  |  Malicious → Take Action

Why Federated Learning for IDS?

Traditional IDS approaches require centralized data sharing, which organizations cannot do due to privacy constraints. Federated Learning (FL) addresses this by allowing collaborative model training without sharing raw data.

  • ✦ Organizations cannot share raw traffic due to privacy regulations
  • ✦ FL allows collaborative model training without sharing data
  • ✦ Requires continuous training with distributed network data

The Vulnerability

Trust becomes the vulnerability. When participants in federated learning are malicious, they can inject backdoors into the global model without being detected — compromising the entire IDS.

03 · Attack Evolution

Threat Model Evolution

The attack model evolved across three phases, from basic poisoning to sophisticated stealth attacks designed to evade defenses.

PHASE 01

Model Poisoning + Replacement

The attacker's goal is to insert a backdoor into the global model. Achieved by manipulating model updates during the federated aggregation round.

Brute-Force Attack
PHASE 02

Adaptive Stealth Attacker

The attacker measures and adjusts the magnitude of their update to blend in with honest clients. Uses "Ninja" logic with L2 projection for geometric camouflage, making updates indistinguishable from benign ones.

Stealth Attack
PHASE 03

PfedBA — Final Threat Model

The most sophisticated model combining personalized federated learning with backdoor injection. Designed to overcome state-of-the-art defenses including FLAME.

Advanced Persistent

Stealth Scaling Strategy

The adaptive attacker employs geometric camouflage — scaling malicious updates so their L2 norm matches honest updates. Standard attacks are "loud" (large weight amplification creates detectable outliers). The stealth approach scales the malicious update norm to match the honest update norm, enabling seamless integration into the aggregation process.

Attack Success Rate (ASR)

ASR measures: of the samples the attacker wants to misclassify, what fraction did the model actually misclassify the way the attacker intended? A lower ASR means the defense is more effective at neutralizing the backdoor.

05 · Methodology

SENTINEL — Our Algorithm

SENTINEL is a novel federated aggregation defense designed to resist both standard and stealth backdoor attacks while preserving NIDS detection accuracy under non-IID data distributions.

Algorithm SENTINEL_Aggregation(client_updates, global_model):
  // Step 1: Compute cosine similarities between updates
  similarities = compute_cosine_similarity(client_updates)
  // Step 2: Cluster updates by behavior
  clusters = cluster_by_similarity(similarities)
  // Step 3: Identify and down-weight suspicious clusters
  weights = assign_trust_weights(clusters, global_model)
  // Step 4: Weighted aggregation
  return weighted_aggregate(client_updates, weights)

Key Design Insight

SENTINEL detects that accuracy alone cannot identify backdoors. A model with 99.95% ASR can still achieve 93% overall accuracy — proving that accuracy metrics hide backdoor success. SENTINEL uses behavioral clustering to identify malicious updates before aggregation.

Non-IID Robustness

Unlike many defenses that degrade under heterogeneous (non-IID) data, SENTINEL is specifically designed to handle Dirichlet-distributed data across clients, making it practical for real-world federated NIDS deployments where network traffic patterns differ across organizations.

Key Result: Accuracy Does Not Reveal Backdoors

A critical finding: a model can achieve 93% classification accuracy while simultaneously having 99.95% ASR. This demonstrates that standard accuracy metrics are insufficient for evaluating federated NIDS security — specialized defense mechanisms like SENTINEL are necessary.

07 · Limitations

Current Limitations

Transparency about the boundaries of our research — areas for future work and improvement.

  • 01

    Single-Feature Trigger

    Our experiments used a single-feature trigger (feature[0] = 5.0). Real-world attacks typically employ multi-feature triggers, which may behave differently against SENTINEL's clustering approach.

  • 02

    Limited Federation Scale

    Tested with only 10 clients. Larger federations with 100+ participants may exhibit significantly different dynamics, and SENTINEL's performance at scale remains unexplored.

  • 03

    Static Trigger Design

    In practice, adversaries may use dynamic or adaptive triggers that change across rounds. SENTINEL was designed against static triggers and may need extension for adaptive adversaries.

  • 04

    PfedBA Vulnerability

    Under the most advanced PfedBA attack with 40% malicious clients, SENTINEL shows reduced robustness. The personalized federated approach of PfedBA represents a harder challenge requiring further investigation.
